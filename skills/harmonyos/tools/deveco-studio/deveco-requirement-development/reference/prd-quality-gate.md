# PRD 质控与定稿阈关（阶段三专用）

> 本文件仅在 `prd-gen.md` 的阶段三生效，用于执行 `§11 / §11.5 / §11.6 / §12`。  
> 目标：在进入编码前，确保 PRD 可执行、可追溯、前后一致。

---

## §11 PRD 验收清单（Review Checklist）

### §11.1 完整性检查

- [ ] `§1~§10` 章节全部存在，无缺失
- [ ] `§3` 每个区域均有颜色/字体，无裸十六进制
- [ ] `§4` 每个交互项均有「触发→中间→结果」三段描述
- [ ] `§4` 跨组件联动已从调研报告 `§10` 迁移完整
- [ ] `§5` 所有断点均有独立描述
- [ ] `§9` 每个接口有完整参数/返回/错误处理/导入路径

### §11.2 规范符合性

- [ ] 无硬编码颜色（十六进制）
- [ ] 图标使用项目图标枚举名，而非 unicode 或路径
- [ ] 时间字段标注单位

### §11.3 接口对齐

- [ ] `§9` 中每个接口名与实际代码/API 文档一致（已通过代码检索或接口文档核对）

### §11.4 可开发性

- [ ] `§10` 待确认事项中无 `[阻塞编码]` 未解决项
- [ ] `§8` 文件结构与调研报告 `§8` 方案论证结论一致

**验收不通过**：返回 `prd-gen.md` 阶段二补充完善，直到所有检查项通过。

---

## §11.5 实现方案技术校准【未完成禁止编码】

验收通过后，对 PRD 的每一个实现决策进行主动校准，发现偏差必须回写 PRD。  
禁止使用“通常应该没问题”类结论替代核实。

校准覆盖六个维度（全部必做）：

- **A 接口校准**：逐一核实 `§9` 每个接口的类名/方法名/参数类型/导入路径
- **B 图标枚举校准**：逐一确认图标枚举值实际存在。按以下优先级执行：
  1. 若存在 `docs/ui-components.md` 或 `rules/ui-components.md`，在文档中查找枚举定义
  2. 若上述文档均不存在，在工程源码中检索枚举定义文件（代码检索方式统一遵循 [common.md §6](common.md)，此处给出示例）：
  3. 若源码检索也无法确认，在 `§10 待确认事项` 中逐个标注 `[阻塞编码] 图标枚举值 {xxx} 未确认，需人工核实`
  - **多个枚举无法确认时**：逐项列出，全部作为独立阻塞项，不合并为一条；编码阶段遇到未确认项时跳过该图标（使用占位符），待确认后补填
- **C 颜色校准**：逐一确认 PRD 中每个颜色引用有效，按以下优先级执行：
  1. 若存在 `docs/ui-colors.md` 或 `rules/ui-colors.md`，在文档中查找对应 Token 名称，确认存在即通过
  2. 若上述文档不存在，在工程源码中检索资源定义（代码检索方式统一遵循 [common.md §6](common.md)，此处给出示例）：
     确认 `$r('app.color.xxx')` 或 `Color.xxx` 枚举值在源码/资源文件中有定义即通过
  3. 若源码检索也无法确认，在 `§10 待确认事项` 中标注 `[阻塞编码] 颜色 Token {xxx} 未确认，需人工核实`
  - **通过标准**：PRD 中无裸十六进制颜色值，且所有颜色 Token / 枚举值均可在文档或源码中找到对应定义
  - **新工程降级（源码尚不存在时）**：若工程刚初始化、资源文件均不存在，本项统一标注 `[阻塞编码] 工程源码未初始化，颜色 Token {xxx} 无法核实，编码前须先完成工程初始化并核实`，继续推进其余校准维度。
- **D 架构层级校准**：确认无反向依赖，路由器/服务注册符合项目规范
- **E 可复用组件核查**：确认拟新建组件是否已有封装；已有则在 `§8` 标注 `[复用]`，而非 `[新增]`
- **F 灰度开关检查**：确认是否需要 Feature Flag；若需要，必须在 `§10` 标注开关策略并获得确认

**六维校准完成后，输出校准结论表**（格式如下，逐维度填写，不得留空）：

```markdown
## §11.5 技术校准结论

| 维度 | 校准项 | 结论 | 备注/风险 |
|------|--------|------|-----------|
| A 接口 | ClassName.method(params) | ✓ 已核实 / ✗ 不存在 / ⚠ 待确认 | 导入路径或问题说明 |
| B 图标 | IconEnum.XXX | ✓ 已核实 / ✗ 不存在 / ⚠ 待确认 | 来源文件或占位说明 |
| C 颜色 | $r('app.color.xxx') | ✓ 已核实 / ✗ 不存在 / ⚠ 待确认 | 来源文件或占位说明 |
| D 架构 | 依赖链路描述 | ✓ 无反向依赖 / ⚠ 有风险 | 风险描述 |
| E 复用 | 拟新建组件名 | ✓ 已有封装 [复用] / ✗ 确需新建 | 复用组件路径或说明 |
| F 灰度 | Feature Flag 策略 | ✓ 不需要 / ✓ 已加入 §10 / ⚠ 待确认 | 开关名称或说明 |
```

> ⚠️ 存在 `✗` 或 `⚠ 待确认` 项时，必须在 PRD `§10` 补充对应 `[阻塞编码]` 条目，禁止带未解决项进入编码。

## §11.6 PRD 全文一致性审查【未完成禁止定稿】

逐章节做交叉验证：

| 检查对 | 检查内容 |
|--------|---------|
| `§3 ↔ §6` | 区域说明中的字段名与数据模型字段名一致 |
| `§4 ↔ §9` | 交互行为中提及的接口与 `§9` 接口映射一一对应 |
| `§5 ↔ §8` | 多设备描述的组件差异与文件结构中的组件数量对应 |
| `§8 ↔ 调研 §8` | 文件结构建议与方案论证结论一致 |
| `§10 → 全文` | 待确认事项已在相关章节中预留占位说明 |

通过标准：
- 输出不一致项清单（可为空）；
- 不一致项已修复或明确阻塞状态。

## §12 设计稿版本管理【必执】

在 PRD 头部记录设计稿版本信息：

```markdown
## 设计稿版本记录

| 版本 | 快照时间 | 链接 | 主要变更 |
|------|---------|------|---------|
| v1.0 | {YYYY-MM-DD HH:MM} | {设计稿URL} | 初版 |
```

设计稿迭代时：追加新行，并在对应章节标注 `[v1.x 新增/变更]`。
