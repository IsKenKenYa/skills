# 代码评审与经验沉淀
---

## 入口条件

- 当前改动范围明确（关联 PRD、issue 或用户描述）
- `coding.md` 验证清单已全部通过（Happy Path / 空态错误态 / 多设备断点 / 交互状态一致性均已验证）
- 静态检查与编译均无报错

**入口条件不满足时的处理规则**：

| 缺少的条件 | 处理方式 |
|-----------|---------|
| 改动范围不明确（无关联 PRD/issue/描述） | 向用户确认本次改动的功能背景与影响范围，补充说明后再开始评审 |
| `coding.md` 验收清单未全部通过 | 告知用户当前哪些验收项未通过，引导先回到 `coding.md` 阶段八完成验收，再进入评审 |
| 静态检查或编译有报错 | 告知用户当前存在报错，禁止在有报错的状态下进行评审；引导先修复报错再回到本流程 |

> 若用户明确知晓风险并要求强行进入评审，可在评审结论顶部注明"⚠️ 入口条件未全部满足（说明缺少项），评审结论仅供参考，不作为合并依据"后继续执行。

## 文档边界（避免与其他阶段重复）

- 编码、编译、推包操作细节、联调与日志定位细节以 `coding.md` 为准
- 本文只负责风险分级评审、性能审视、经验沉淀与回归建议

---

## 阶段一：自动化与规范扫描

1. 对本次改动涉及的文件运行项目规定的静态分析 / Lint / 格式化检查
2. 对照 `rules/code-style.md`（若存在）做命名、注释、导入规范扫描
3. 若有 CI，提示用户在本地或通过流水线确认通过

---

## 阶段二：分级检查清单

### P0（必须修复才能合并）

- 空指针 / 未初始化状态可能导致 Crash
- 敏感信息明文日志、硬编码密钥
- 主路径业务逻辑错误或可预见的数据损坏
- 违反工程明确禁止项（如`any`、关键异步未处理等）

### P1（强烈建议本 PR 内修复）

- 明显性能问题：主线程重计算、大列表无复用/无懒加载等可预见卡顿
- 错误与加载状态无用户可感知反馈
- 与 PRD §4 明显不一致的交互

### P2（可建 follow-up）

- 命名与可读性、注释补充、小范围重构建议

**若存在 P0/P1**：回到 `coding.md` 修复后重新静态检查与推包，再回到本评审流程。

---

## 阶段三：对照模块已知问题

若工程内存在 `/docs/ai-coding-issues.md` 或 `memory/bugs.md`，则读取并核对本次改动是否触碰已知坑位；**若均不存在**，本阶段记为已跳过，并在评审结论中注明"未找到已知问题文档，建议后续建立 `memory/bugs.md` 积累模块坑位记录"。

---

## 阶段四：性能检查（五项）

逐项给出结论（通过 / 风险 / 需真机复测），并按以下规则对应风险等级：

- **风险**（可预见的明显性能劣化，如大列表无复用、主线程阻塞）→ 等同 **P1**，强烈建议本 PR 内修复；若需延后，必须建 follow-up 并在评审结论中注明。
- **需真机复测**（动画帧率、偶现卡顿等无法静态判断的问题）→ 在评审结论中注明，由用户决定是否在合并前完成复测。
- **通过** → 无需额外处理。

| 维度 | 检查要点 |
|------|----------|
| 渲染 | 大列表是否分页或虚拟化；是否频繁全量状态刷新 |
| 内存 | 大图、监听、定时器是否在页面销毁时释放 |
| 首屏 | 关键路径是否有不必要阻塞 |
| 网络 | 重复请求、超时与重试策略 |
| 动画 | 帧率与掉帧风险；弹层与导航顺序是否安全 |

**若存在性能风险项（P1）**：回到 `coding.md` 修复后重新静态检查与推包，再回到本评审流程（与阶段二 P1 处理路径一致）。

---

## 阶段五：多设备与断点布局评审

> 若本次需求 PRD §5 明确了多设备适配要求，本阶段必执；若 PRD §5 明确标注"仅手机"，可跳过并在评审结论中注明。

逐项给出结论（通过 / 风险 / 需真机复测）：

| 维度 | 检查要点 | 风险等级 |
|------|---------|---------|
| 断点响应 | sm/md/lg 断点切换时布局是否正确重排，无溢出/重叠 | P1（可预见） |
| 折叠屏展开/折叠 | 内外屏切换时组件是否正确销毁重建，状态是否保留 | P1（若适用） |
| 平板大屏 | 列数/组件尺寸是否按 PRD §5 描述随断点变化 | P1（若适用） |
| 热区与触控 | 在大屏设备上点击热区是否仍符合最小 44×44 vp 要求 | P2 |
| 字体缩放 | 系统大字体模式下文本是否溢出容器或被截断 | P1（可预见） |

**若存在断点布局风险（P1）**：回到 `coding.md` 修复后重新静态检查与推包，再回到本评审流程。

---

## 阶段六：经验沉淀

经验沉淀按 [common.md §4](common.md) 执行（含 `memory/` 主动创建规则）。

### Step 1：ArkTS 平台差异写入 memory/arkts-patterns.md

评审阶段二～五中，若发现以下任一情况，**必须**将该差异写入 `memory/arkts-patterns.md`（按 PATTERN 模板追加）：
- 代码中存在 ArkTS 特有的错误做法（哪怕已修复）
- 发现与常规 TypeScript / Web 行为不同的平台特性
- 评审时补充的 P0/P1 涉及 ArkTS 框架约束

写入后，执行 Step 2 的回写判定。

### Step 2：新坑回写 coding.md 高发约束表判定

对本次新写入的每条 `PATTERN-{序号}` 条目，按以下标准判断是否需要回写 `coding.md`：

| 判断标准 | 处理方式 |
|---------|---------|
| 该模式差异**仅在本项目特定业务逻辑中出现**，不具备通用性 | 只写 `memory/arkts-patterns.md`，`PATTERN` 条目末行标注"是否已回写 coding.md：否（项目特定）" |
| 该模式差异**在任何 ArkTS 工程中均可能踩坑**（如 API 使用约束、框架行为差异） | **必须回写**：在 `coding.md` "ArkTS / HarmonyOS 平台特有高发约束"表格末尾追加一行，格式与现有行一致（场景 / 错误做法 / 正确做法），`PATTERN` 条目末行标注"是否已回写 coding.md：是（见 coding.md 第 N 行）" |
| 不确定 | 默认视为通用，执行回写，并在 `PATTERN` 条目备注"通用性待确认" |

> 回写 `coding.md` 后无需等待用户确认，直接在评审结论中注明"已追加 N 条 ArkTS 约束到 coding.md"。

### Step 3：其他经验沉淀

编码阶段（`coding.md` 阶段七 Step 4）已向 `memory/bugs.md` / `memory/crashes.md` / `memory/hard-problems.md` 写入过问题记录；评审阶段**禁止重复追加**已存在条目，必须先执行去重判定：

1. **去重判定**：在写入前，先在对应 memory 文件中按**模块名 + 功能关键词 + 错误现象**搜索是否已有同类条目：
   - 已存在同类条目 → 在原条目末尾追加"评审补充：{新发现信息}"，**不新增条目**
   - 不存在同类条目 → 按对应模板新增条目（视为漏记，补写）
   - memory 文件不存在 → 按 [common.md §4.1](common.md) 主动创建后再写入

2. **可写入评审阶段的内容类型**（满足任一则需写入，否则跳过本步骤）：
   - 评审阶段二～四中新发现的、编码阶段未记录的 Bug / Crash
   - P0/P1 问题中涉及的根因分析，编码记录中未填写"根因"字段时补全

3. 评审中归纳的通用重构建议（P2 级）若不写入代码，在对话中输出，由用户决定是否写入 `memory/`

---

## 完成条件

- 无未关闭的 P0/P1
- 性能五项已记录结论
- 经验沉淀已写入 `memory/`（按 common.md §4 主动创建规则执行，只读环境除外）
- 通用 ArkTS 模式差异已完成 Step 2 回写判定，结论已在 `PATTERN` 条目中标注
