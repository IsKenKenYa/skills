# 设计稿解析与需求文档生成

> 通用规则（可选目录、复用优先、重试与沉淀）统一遵循 [common.md](common.md)。
> 本文聚焦调研、设计稿与 PRD 产出；编码/联调/评审细节分别遵循 `coding.md`、`review.md`。
> `§11 / §11.5 / §11.6 / §12` 的详细阈关清单统一在 [prd-quality-gate.md](prd-quality-gate.md)。

---

## 阶段零：需求调研

> 用户提供粗略需求后，优先进行调研，完成调研报告并经用户确认，再进入阶段一解析设计稿。禁止跳过本阶段直接写 PRD。

### Step 1：确认调研范围

收到粗略需求后，向用户确认：
- 所属业务领域和功能模块名称
- 在项目中的层级位置（顶部导航 / 首页 / 内容流 / 设置等）
- **目标设备范围**：手机 / 平板 / 折叠屏 / 多端通用（直接影响 §5 多设备适配工作量，必须在调研阶段确认）
- 是否已有相关历史变更或问题单可参考

### Step 2：执行调研

并行执行以下调研，不得仅凭上下文记忆回答：

**① 模块接口/依赖能力盘点**：在 `oh_modules/`、`packages/` 下搜索相关接口模块（代码检索遵循 [common.md §6](common.md)），仅读取与本需求领域最相关的 1～3 个入口文件（`index.ets` / `index.d.ts`）。

> **目录不存在的降级处理**：若 `oh_modules/` 与 `packages/` 均不存在（如全新初始化工程），则本项标注"依赖目录尚未初始化，无法盘点现有接口模块"，并在调研报告 `## 5. 架构约束与风险` 中补充说明"工程依赖未初始化，编码前须先执行 `ohpm install` 或等效初始化命令"，继续推进其余调研步骤。

**② 本模块现有实现调研**：搜索相似功能/组件/工具函数，重点确认可复用项与需改造的模块边界。

```powershell
# 跨平台推荐：优先 rg 检索（列举接口包目录名，不读文件内容）
rg "_interface" .\oh_modules\ --files
rg "_interface" .\packages\ --files
```

重点关注与本需求相关的领域（如推荐、内容、AI 服务等）接口包，读取其入口文件了解可调用的能力范围。

**③ 架构层级与依赖关系梳理**：使用任意"文件读取能力"读取 `package.json` / `oh-package.json5`，确认以下两点：
- 本次需要依赖的接口模块已在配置中声明
- 不存在反向依赖风险（下层模块不得依赖上层）

**④ 现有交互行为盘点【改造类需求必做，新增页面可跳过】**

> **改造类 vs 新增页面判断标准**（满足任一"改造类"条件即视为改造类，必须执行本项）：
>
> | 判断维度 | 改造类（必做） | 新增页面（可跳过） |
> |---------|-------------|----------------|
> | 是否修改已有文件 | 修改 1 个以上已有 `.ets` 文件 | 全部新建文件，不动任何已有文件 |
> | 是否影响公共组件 | 改动被多处引用的组件或工具函数 | 完全独立的新页面，无公共组件改动 |
> | 是否有跨组件联动 | 新功能改变其他组件的状态/回调/显示规则 | 新页面与现有页面无状态交叉 |
>
> 若判断存在争议，**优先视为改造类执行**，宁可多盘点不漏。

> **目的**：PRD 描述的是「新增的东西」，但现有代码里已有的视觉状态和联动规则默认被忽略，
> 导致新功能写完后这些规则被意外破坏，或者本应保留的动画/显示逻辑从未写进 PRD。

对本次需要改造的每个文件，**读取其当前实现**，逐项列出：

| 盘点项 | 需要记录的内容 |
|--------|--------------|
| **现有视觉状态** | 该组件有哪些显示/隐藏状态，分别由哪个状态字段控制 |
| **现有动画/动效** | 已有的动画（浮动文字、点赞缩放、滑入等），新 PRD 是否需要保留 |
| **现有 callback 签名** | 组件已接受哪些 `onXxx` 回调，父页面已绑定哪些 |
| **现有条件渲染** | 哪些区域在特定条件下隐藏，新功能是否影响这些条件 |
| **现有层叠/点击区** | 是否有 z-index 或点击穿透设置，新功能是否引发冲突 |

盘点结果写入调研报告 `## 10. 现有交互行为清单`：

```markdown
## 10. 现有交互行为清单

| 文件 | 现有行为 | 新 PRD 处理方式 |
|------|---------|----------------|
| Foo.ets | fieldX 在小屏评论打开时无隐藏逻辑 | 新增：showPanel=true 时隐藏 |
| Bar.ets | likeAnimation 已实现 | 保留：PRD §2.6 需明确说明保留此动画 |
```

> ⚠️ 此清单在阶段二生成 PRD 时，直接作为 §4.x 跨组件状态联动的输入素材。

## 阶段一：获取与解析设计稿

若用户未提供设计稿链接，先询问。

**直接提供设计稿链接的快速入口**：若用户在初始消息中已包含设计稿链接（Figma / MasterGo / Sketch 等 URL），可跳过 Step 4 的链接询问，直接在阶段零调研完成后进入本阶段解析。阶段零的调研报告仍需完整输出，不得省略。

**无设计稿降级路径**：用户明确无设计稿时：
1. 跳过读取，阶段结论注明：`UI 信息来源：无设计稿，依现有代码调研 + 用户口述推断`
2. PRD §2 线框图标注 `[无设计稿，线框图为推断结果，需用户确认]`
3. PRD §3 所有尺寸/颜色标注 `[待设计稿确认]`，并在 §10 补充 `[阻塞编码]` 项
4. 流程继续推进，进入编码前阻塞项必须解决

用户提供链接后，调用对应设计工具 MCP 获取 DSL 数据，提取：页面名称/尺寸/背景色、组件层级树、各区域尺寸/间距/字体/颜色、交互热区与事件描述。

## 阶段一-A：方案论证与可行性确认

> 设计稿解析完毕后，禁止直接写 PRD。必须先进行方案论证并经用户确认。

### Step 1：AI 主动输出初步方案

设计稿解析完成后，结合调研报告，主动对以下关键决策给出方案和理由：

| 决策维度 | 必须覆盖的问题 |
|---------|------------------|
| 架构选型 | 新功能属于哪个层级？新增页面还是改造现有组件？是否需要新接口模块？ |
| 状态管理 | ViewModel/Store 是共享还是页面独立？数据流向是什么？ |
| 组件选型 | 关键 UI 区域用哪个现有组件实现，为什么这样选 |
| 互动边界 | 某个交互的触发条件、进入点、退出方式具体是什么 |
| 性能边界 | 列表多大用虚拟列表？页面重用情况如何处理？ |
| 多设备适配 | 折叠屏/外屏/平板与手机布局差异点在哪里（若适用） |

**输出格式**：

```
初步方案概要

1. 架构层面：{...}
   → 理由：{...}

2. 主要组件：{...}
   → 选型依据：{...}

3. 关键交互边界：{...}

4. 不确定项：{候选方案A / 候选方案B，各自利弊}
```

### Step 2：反复论证对话

**AI 必须给出明确结论，不得回答「应该可以」「大概是」**。每个回答包含：结论（明确选择或边界）、依据（代码证据或框架规范）、风险（如有不确定性）。

- 结论：明确的选择或边界
- 依据：代码证据（文件:Line）或框架/平台规范依据
- 风险：如果有反例或不确定性，明确说明

每个确认的决策**就地记录**到调研报告的 `## 8. 方案论证结论` 表格中（模板见调研报告 Step 3 输出的 `{FeatureName}_investigation.md`）。示例行：

```
| 是否新增 ViewModel | 页面独立 | XxxPage 已有共享 VM 且字段过多 | @开发者 |
| 组件选型 | 用底部面板组件 | 嵌套动画需要目前 API | @开发者 |
```

### Step 3：Demo 验证（不确定项必须执行）

对 UI 交互/动效类不确定项，制作最小化 Demo：
1. 路径优先级：本次需求模块目录下 `demo/{FeatureName}Demo.ets` → 工程入口 `entry/src/main/ets/demo/{FeatureName}Demo.ets`
2. 只包含待验证部分，不写完整页面
3. 编译推包，失败重试上限 2 次；2 次后仍失败，执行以下步骤：
   - 在调研报告 §9 记录失败原因，状态标为 `[待人工验证]`
   - 在调研报告 §8 对应决策行填写 `⚠ 待人工验证（Demo 编译/运行失败，见 §9）`
   - **阻塞判定**：若该不确定项直接影响 PRD 核心交互（§4）或架构选型（§8），则**阻塞推进**，进入 Step 5 向用户列出阻塞原因，等待用户给出结论（人工评审结果 / 接受风险继续 / 调整方案）后再推进；若不确定项仅影响动效细节，可**带风险继续**，在 PRD §10 追加 `[阻塞编码] Demo 未验证通过：{问题描述}，编码前须人工确认`
4. Demo 成功运行时：人工评审呈现效果，结论写回 §8，标注是否通过 Demo 验证
5. **验证完成后立即删除 Demo 文件**，不得提交仓库

> 问题可通过现有代码或框架文档明确回答时，无需 Demo。

### Step 4：记录发现的问题

论证中发现的风险/约束写入调研报告 `## 9. 已记录问题`（格式：编号 | 问题描述 | 影响范围 | 处理方式 | 状态）。

### Step 5：用户确认进入编档

列出全部状态为"待确认"的 Q-00x 问题，等待用户回复后，才进入阶段二。

## 阶段一-B：读取并刷新/创建当前工程 UI Pattern（生成 PRD 前必做）

> **定位说明**：本阶段在阶段一-A 方案论证完成、进入阶段二写 PRD **之前**执行，是阶段二的强制前置准备步骤。虽然编号沿用"一-B"以表示其紧接设计稿解析之后，但执行时机为：阶段一-A Step 5 用户确认完成 → 立即执行本阶段 → 再进入阶段二生成 PRD。
> 进入阶段二前，必须先读取工程内已有 Pattern 文档；缺失或明显过期时先刷新/创建再写 PRD。

### Step 0：按需触发判断（优先执行，决定本阶段执行深度）

根据本次需求特征，判断需要执行的深度：

| 需求特征 | 判断标准 | 执行深度 |
|---------|---------|---------|
| **仅复用已有组件**，不新增任何 UI 文件 | §8 文件结构中无 `[新增]` UI 组件/页面文件 | **轻量模式**：只执行 Step 1（读取 Pattern 文档），跳过 Step 2～4；直接进入 Step 5 |
| **新增全新 UI 类型**，或 Pattern 文档不存在 | §8 中出现工程内未见过的 UI 类型（新页面模板、新弹层类型等），或首次建立 Pattern | **完整模式**：执行全部 Step 1～4 |


### Step 1：定位 Pattern 文档

按优先级查找（有则读）：`/rules/ui-patterns.md` → `/docs/ui-patterns.md`

### Step 2：查找工程中的 UI 组件并生成候选清单

扫描 `/src/**/pages/` 和 `/src/**/components/` 下的 `*.ets` 文件（**仅获取路径，不读内容**）。

对比实际文件列表与 `ui-patterns.md` 的 `**参考文件**` 清单，输出 Delta 清单（新增文件 / 已覆盖文件）。

价值筛选（仅凭文件名判断，不读内容）：
- 收录：含 `Page`、`Card`、`Panel`、`Dialog`、`Sheet`、`Item`、`Banner` 等 UI 语义词
- 跳过：含 `ViewModel`、`Model`、`Util`、`Helper`、`Service`、`Controller` 等非 UI 语义词

> **扫描说明**：本步骤使用目录列举工具（`list_dir` / `glob` / 文件树能力）**只获取文件名与路径，不读取文件内容**。单次列举最多保留 **60 条** `*.ets` 路径；超出 60 条时，按以下优先级截断：
> 1. 与本次需求所属模块路径前缀相同的文件（全部保留）
> 2. 其余按路径字母序取前若干条，凑满 60 条
> 3. 被截断的文件路径统一记录为"未扫描"并在 Delta 清单末尾注明数量
>
> 截断判定由 AI 在获得文件列表后手动筛选，不依赖工具自动过滤。

执行动作：

1. 调用目录列举能力获取上述两个目录下全部 `*.ets` 文件路径列表（**仅路径，不读内容**）。
2. 按上方截断规则将列表控制在 60 条以内，得到"当前实际 UI 文件列表"。
3. 将实际 UI 文件列表与已读取 `ui-patterns.md` 的 `**参考文件**` 清单做差异对比。
4. 输出 Delta 清单：
   - 新增文件：实际存在但 Pattern 未覆盖（候选新增 Pattern）
   - 已覆盖文件：进入抽查，判断是否需要刷新已有 Pattern
5. 对每个新增文件做价值筛选（决定是否收录），**此步骤仍不读取文件内容，仅凭文件名与路径判断**：
   - 值得收录：文件名含 `Page`、`Card`、`Panel`、`Dialog`、`Sheet`、`Item`、`Banner` 等 UI 语义词
   - 可跳过：文件名含 `ViewModel`、`Model`、`Util`、`Helper`、`Service`、`Controller` 等非 UI 语义词
6. 对价值筛选后拟收录的文件**逐个读取内容**，提取：关键词、核心布局骨架、样式规律（尺寸/间距/颜色/动画时长）。**读取文件总数不超过 20 个**（超出时优先读与本次需求模块最相关的文件）。

### Step 3：判断是否需要刷新/创建

**快速退出条件（满足全部以下条件时，跳过 Step 4，直接进入 Step 5）**：
- Pattern 文档已存在（`/rules/ui-patterns.md` 或 `/docs/ui-patterns.md`）
- Delta 清单中新增文件数量为 0（无未覆盖的 UI 文件）
- Phase 1 抽查的已引用文件与文档描述无重要差异（根容器/布局属性/样式数值无变化）

满足快速退出条件时，向用户报告"UI Pattern 基线已是最新，无需刷新"，直接跳至 Step 5。

**需执行刷新的条件**：任一情况需执行 Step 4：① 未找到 Pattern 文档；② 与源码差异较大（新增文件未覆盖或关键布局片段过期）。

### Step 4：执行刷新/创建


#### Phase 1 — 检查已引用文件的变更

对文档中已引用文件按以下规则抽查：

- **优先抽查**：与本次需求直接相关的组件/页面文件（最多 5 个）
- **次优抽查**：最近被修改过（git log 最新提交涉及）的已引用文件（最多 3 个）
- **上限**：单次 Phase 1 最多抽查 **8 个**文件；超出时以"与本需求相关性"降序排列，取前 8 个

抽查步骤：
1. 读取实际文件
2. 对比文档片段，判断根容器、关键布局属性、样式规律数值是否变化
3. 如有重要差异，更新对应 Pattern 的代码片段和样式规律

#### Phase 2 — 写入 `ui-patterns.md`

按以下规则选择写入章节：

**优先读取工程已有章节结构**：写入前先读取 `ui-patterns.md` 文件头部，提取其中已有的 `## N. xxx` 章节列表，以工程实际章节为准进行匹配。仅当工程文件不存在（新建场景）时，才使用下方默认分类表作为初始章节结构。

| 新 Pattern 类型 | 放入章节（工程无对应章节时的默认值） |
|---|---|
| 页面级（全屏页、导航目标页） | `## 1. 页面模板` |
| 卡片类（Feed / Grid Item） | `## 2. 卡片组件` |
| 弹层/面板/抽屉 | `## 3. 弹层/面板` |
| 导航栏/操作栏/TabBar | `## 4. 导航/操作组件` |
| 空态/错误态/Loading | `## 5. 状态视图` |
| 列表 Item | `## 6. 列表子项` |
| 输入框/表单 | `## 7. 输入与表单组件` |
| 不确定 | 最相近章节末尾 |

**工程章节不存在时的降级处理**：若新 Pattern 类型在工程已有章节中找不到对应匹配项，按以下步骤处理：
1. 在工程 `ui-patterns.md` 末尾新增章节（章节号续接当前最大编号），标题以工程现有命名风格为准（中文/英文/混合均与已有保持一致）
2. 在变更摘要中注明"新增章节：`## N. xxx`（原分类表无对应项，以工程实际 UI 类型命名）"
3. 禁止将 Pattern 强行塞入语义不匹配的已有章节

写入模板：

```markdown
### X.N 标题
**关键词**：词1、词2、...
**参考文件**：`路径/文件名.ets`（重点读 `函数名`）
// 核心布局骨架（省略业务细节用 ... 代替）
**样式规律**：关键数值简短描述
```

执行要点：仅修改 `ui-patterns.md`，不改源码；Phase 1 后向用户报告 Delta 概览，无需等待确认；单次最多新增 10 个 Pattern；完成后输出变更摘要。

### Step 5：Pattern 作为阶段二输入约束

- §1：优先复用已有布局骨架
- §2：优先沿用已有样式规律；需偏离时说明原因
- §3：优先对齐已有交互范式
- §4：已有可复用组件标注 `[复用 {组件名}]`

### Step 6：无 Pattern 降级

PRD 头部注明 `UI Pattern 基线：未获得可用 ui-patterns 文档，本次按设计稿与现有代码调研结果制定`，并在 §10 补充 `[阻塞编码] 是否补建模块 UI Pattern 基线文档`。

## 阶段二：生成需求文档

保存路径：`/docs/prd/{FeatureName}_requirements.md`

文件头部：
```markdown
# {功能名称} 需求文档
设计稿来源：[{Design} - {功能名}]({设计稿URL})
代码规范遵循（若工程内存在则引用路径）：`/rules/code-style.md`
```

核心章节目录（1~10，顺序固定，禁止省略）：
```
§1 功能概览  §2 整体布局（含 ASCII 线框图）  §3 区域详细说明
§4 交互行为  §5 多设备适配  §6 数据模型
§7 代码规范约束  §8 文件结构建议  §9 接口映射  §10 待确认事项
```

> `§11 / §11.5 / §11.6 / §12` 为质控章节，在阶段三按 [prd-quality-gate.md](prd-quality-gate.md) 执行。

### 逐章节填写规范

**§1**：一句话功能定义 + 核心用户价值。

**§2**：必须提供 ASCII 线框图，标注各区域名称、高度/比例。

**§3**：逐区域展开，每个区域包含：尺寸/边距（设计稿精确值）、字体（设计系统变量名）、颜色（禁止裸十六进制，用 Token）、图标（项目枚举值，需代码检索确认存在）、显示/隐藏条件。

**§4**：每个交互项格式：
```
**[交互名称]**
- 触发条件：{...}  - 中间状态：{Loading/动效}
- 最终结果：{成功/失败/边界}  - 跨组件联动：{来源于调研报告 §10}
```

**§5**：按断点（sm/md/lg）逐档描述列数/组件大小/显示隐藏策略；折叠屏展开/折叠状态（若适用）。

**§6**：每个字段附 JSDoc 注释 + 数据来源（接口字段名），时间字段标注单位（ms/s）。

**§7**：列出本功能需特别注意的规范点（引用 `code-style.md` 对应章节）。

**§8**：树形列出新增/修改文件，标注 `[新增]` / `[修改]` / `[复用 {组件名}]`，并补充实现责任矩阵（文件 → 负责人 → 状态）。

**§9**：每个接口格式：方法（`ClassName.method(params)`）、参数、返回、错误处理（失败时 UI 行为）、导入路径。

**§10**：阻塞性问题标记 `[阻塞编码]`，须在进入编码前由用户确认。

## 阶段三：需求文档验收

需求文档生成后，**必须按顺序执行以下全部步骤，禁止跳步**：

| 顺序 | 步骤 | 跳过后果 |
|------|------|----------|
| 0 | §8 实现责任矩阵填写 | 联调发现缺功能 |
| 1 | §11 PRD 验收清单 | 编码期间发现缺漏，返工成本高 |
| 2 | §11.5 实现方案技术校准 | 编码用错 API，运行时 Crash |
| 3 | §11.6 PRD 全文一致性审查 | 文档前后矛盾，编码期间反复对齐 |
| 4 | §12 设计稿版本记录 | 设计稿迭代后无法追溯影响范围 |

> ⚠️ **步骤 1～4 完整检查项与通过标准均在 [`prd-quality-gate.md`](prd-quality-gate.md) 中定义，必须打开该文件逐条执行，不得自行判定通过。**

执行顺序：先完成 §8 责任矩阵 → 打开 `prd-quality-gate.md` 依次完成 §11 → §11.5 → §11.6 → §12 → 任一项未通过返回阶段二补齐后重跑。
