# 编码实现、联调修复与验收

> 本阶段支持两类入口：
> 1) **新功能/结构改动**：需在 PRD 已定稿（见 `prd-gen.md`）后执行；
> 2) **Bug/联调/快修**：可直接进入修复流程，但需先确认 issue/复现步骤/预期行为。

## 工具前置条件（编码/联调阶段）

本阶段执行前，按 [common.md](common.md) 的「0. 工具前置条件（限定生效范围）」完成环境检查与 MCP 配置确认。
若 `check_ets_files` / `build_project` / `start_app` 等关键工具不可用，按 [common.md](common.md) 的「0.3 工具能力映射与“缺工具先装 MCP”策略」先安装/补充 MCP 并重试原流程

## 文档边界（避免与其他阶段重复）

- PRD 生成、设计稿解析、方案论证以 `prd-gen.md` 为准
- 本文聚焦编码、构建、联调、疑难定位与验收回归
- 风险分级评审与经验复盘以 `review.md` 为准

## 阶段一：编码前准备

### 1.1 规范文件检查（存在则读，不存在跳过并说明影响）

在开始任何 Coding 之前，执行以下“存在则读，不存在跳过并说明影响”检查：

1. **读模块规范**（路径均在用户工程内，**有则读、无则跳过**）：
   - 优先：`rules/code-style.md` 或 `docs/coding-standards.md` 或其他近似文档
   - 按需：`rules/middleware.md` 或 `docs/middleware-guide.md` 或其他近似文档
   - 按需：`rules/ui-components.md` 或 `docs/ui-components.md` 或其他近似文档
2. **读代码模板**：若工程内存在 `templates/`，**仅读 1 个**最匹配模板；否则按官方组件范式手写
3. **读 UI Pattern 索引**：若 `docs/ui-patterns.md` 存在且与本次 UI 相关，则读；若无该文件则跳过
4. **接口/实现分离**：确认是否需要独立接口模块（`*_interface` 或等效结构），外部模块只依赖接口层
5. **【强制】先查内部封装，再动手写**：在当前工程下搜索相似功能、组件、工具函数。重点搜索：加载态/空态/错误态组件、格式化工具函数、可复用 ViewModel 基类

说明要求：任一规范/模板/Pattern 文件不存在时，需在阶段结论明确“未加载文件项 + 可能影响”（例如：代码风格一致性、复用判断准确度、UI 细节对齐度）。

### 1.2 配置文件变更检查

若本次实现涉及以下任一操作，**必须先执行工程同步初始化**，再开始 Coding：

- 新增模块依赖（修改包管理配置文件，如 `package.json` / `oh-package.json5` 等）
- 新增路由页面（修改路由配置文件）
- 新增服务注册（修改应用清单文件）
- 新增本地依赖（修改本地依赖配置文件）

执行方式视项目构建工具而定（如 `project_sync`、`hvigor sync` 等）。

## 阶段二：代码生成

按需求文档 §8 的文件结构**分三批**生成，**每批完成后立即执行静态检查**，不要攒到最后一起查。

### 第一批：骨架 + 数据模型 + 路由注册

1. `model/` — 数据模型（interface / enum / type），每个字段加 JSDoc 注释
2. 路由注册 / 服务注册（修改相应配置文件）
3. 页面/组件文件骨架（空结构，只建文件、写组件基础结构和空 build/render）

→ **立即执行**静态检查，通过后才进入第二批。

### 第二批：核心业务逻辑

4. `viewmodel/` 或 `controller/` — 状态管理（RPC/API 调用、响应式状态）
5. 核心子组件逻辑（数据绑定、事件回调）

→ **立即执行**静态检查，通过后才进入第三批。

### 第三批：UI 细节 + 异常态

6. `components/` — UI 细节（骨架屏、空态、错误态、动效）
7. `pages/` — 页面容器（多设备适配、断点响应式布局）

→ **立即执行**静态检查，全部通过后进入阶段三编译。

> **注意**：第一批骨架阶段若发现需要新增依赖/路由/服务注册，先执行同步再继续写代码。

### 编码硬性约束（违反视为编译前错误）

- 禁止使用 `any` 类型
- 禁止硬编码颜色值
- 禁止硬编码事件字符串（定义为导出常量）
- 禁止直接在全局作用域调用动画 API（使用组件实例或上下文安全的方式）
- 弹层/覆盖层关闭与页面跳转须注意顺序（先关浮层再导航，避免页面栈异常）

### ArkTS / HarmonyOS 平台特有高发约束

以下为鸿蒙开发高频踩坑点，编写代码时**主动预防**，评审时同步检查：

| 场景 | 错误做法 | 正确做法 |
|------|---------|---------|
| `@State` / `@Link` 嵌套对象属性更新 | 直接修改嵌套属性（`this.obj.field = x`），UI 不刷新 | 赋值整个对象或用 `@Observed` + `@ObjectLink` 拆分 |
| `LazyForEach` 数据源 | 传普通数组 | 数据源类必须实现 `IDataSource` 接口，包含 `totalCount()` 与 `getData()` |
| Worker 线程使用 UI 组件 | 在 Worker 中构造 `@Component` 或操作 `@State` | UI 组件与状态操作只能在主线程执行 |
| `@Builder` 函数传参 | 传引用类型后直接修改，触发意外刷新 | 通过 `@BuilderParam` 或明确使用 `@Prop` 单向传递 |
| 列表虚拟化 | 大列表直接用 `ForEach` 全量渲染 | 超过 50 条用 `LazyForEach` + `cachedCount` |
| 资源引用 | 硬编码字符串路径（`"/resources/xxx.png"`） | 统一使用 `$r('app.media.xxx')` / `$r('app.string.xxx')` |
| 页面生命周期副作用 | 在 `aboutToAppear` 中发起未取消的异步任务 | 在 `aboutToDisappear` 中取消订阅/定时器/监听 |
| `router.pushUrl` 异常 | 未处理路由跳转失败（目标页未注册） | 使用 `try/catch` 包裹并在 `§9` 接口映射中注明导入路径 |

## 阶段三：静态语法检查与修复

Coding 完成后，立即调用静态语法检查工具 `check_ets_files` 对所有新增/修改的源码文件进行检查：

- 按检查结果逐条修复，每次修复后重新运行检查
- 同一问题若连续 **3 轮迭代仍未修复**：
  1. 调用平台/框架知识库（如 `harmonyos_knowledge_search`、框架官方文档）查询解决方案
  2. 同时查阅 `/docs/ai-coding-issues.md`（**若存在**）与工程内 `memory/arkts-patterns.md`（**若存在**）是否有相似的差异案例
  3. 记录与沉淀遵循 [common.md](common.md) 的知识沉淀策略

### 常见静态检查问题速查（通用）

| 错误模式 | 解决方向 |
|---------|---------|
| 响应式状态嵌套属性类型推导失败 | 提取扁平状态字段，手动赋值 |
| 组件 API 不存在 | 确认 API 版本，补全 import |
| 动画 API 上下文错误 | 使用组件实例上下文调用 |
| `bigint` / 数字类型拼接报错 | 先 `.toString()` 再拼接 |

## 阶段四：编译构建

静态检查无报错后，调用构建工具 `build_project` 进行完整编译。

- 编译失败 → 根据错误信息修复，重新编译
- **编译重试上限：3 次**（计数含义：针对同一次编码产出，从第一次编译失败起计，每次"修复后重新编译"算 1 次；编译成功后计数清零）
- 若同一编译错误在连续 2 次重试中仍出现，说明修复方向有误，需重新分析根因
- **注意**：阶段四的编译重试计数与阶段六/七的"用户反馈修复计数"相互独立，互不叠加；编译成功后阶段四计数清零，不影响阶段六的起始计数。
- 编译成功 → 进入推包

## 阶段五：推包部署

编译成功后，将应用包推送到模拟器/真机：

- 使用 `start_app`进行推包，推包完成后，**向用户展示功能入口操作路径**（参照需求文档 §4 用户操作方式），等待用户进行功能验收反馈

## 阶段六：问题修复（用户反馈后）

收到用户反馈后，首先确认问题描述足够具体。如果用户描述不足，请求按以下格式补充：

```
问题描述：[具体说明看到的现象，如「点击发布后列表未刷新」]
复现步骤：[一步步操作路径]
预期结果：[PRD §4 中描述的正确行为]
实际结果：[当前实际表现]
截图/录屏：[如有]
```

> 描述越具体，定位越快。不要只说「没生效」，说清楚「做了什么、应该看到什么、实际看到什么」。

明确问题后：

1. **检索历史经验**（按 [common.md](common.md) 执行，有则检索，无则跳过）：
   - 功能/逻辑问题：检索 `memory/bugs.md`
   - Crash/崩溃：检索 `memory/crashes.md`
   - 疑难偶现：检索 `memory/hard-problems.md`
   在每个已存在文件中按关键词（模块名、功能名、错误现象）搜索
2. **最小修复原则**：仅改与问题直接相关的代码，不做额外重构
3. 修复后按顺序执行：**静态检查 → 编译 → 推包**（与本文件阶段三～五一致）

---

## 阶段七：疑难问题定位（阶段六累计修复 3 次后仍存在）

> **修复次数计算规则（全局唯一定义）**：
> - 计数起点：阶段六收到用户首次反馈时，计数器从 0 开始。
> - 计数单位：每完成一次"代码修改 → 静态检查 → 编译 → 推包"完整循环算 1 次，无论修复是否成功。
> - 进入阶段七后的修复同样纳入此计数，不单独重置。
> - 阶段四的编译重试（编码初期产出，编译成功后计数清零）不纳入此计数。
> - 强制挂起门控：累计修复次数超过 5 次（含阶段七内修复）时强制挂起，通知用户人工介入。

若同一问题连续修复 **3 次**后仍未解决（即完成第 3 次"修改→检查→编译→推包"循环后问题仍存在），进入以下流程：

### Step 1：增加日志打点

在关键路径添加项目日志工具打点（使用项目实际日志 API）：

- 数据加载回调处
- 状态变量赋值处
- 条件分支入口处

### Step 2：获取运行日志

通过平台日志工具获取对应模块日志，过滤 TAG 分析根因：

`get_hilog_or_faultlog_recent`（若已配置），按 TAG 过滤

### Step 3：查询知识库

调用框架/平台知识库（如 `harmonyos_knowledge_search`、框架官方文档）查询该问题场景的解决方案，重点搜索以下关键词组合：

- 错误信息原文 +场景关键词
- 组件名 + 平台名 + 场景描述

同时查阅模块内已知问题文档（`docs/ai-coding-issues.md`，若存在）是否有相似案例。

### Step 4：记录疑难问题并判断是否人工升级

问题沉淀按 [common.md §4](common.md) 执行（含 `memory/` 主动创建与各文件初始化模板）。根据问题类型选择对应文件：

| 问题类型 | 目标文件 | 条目前缀 |
|---------|---------|---------|
| 功能/逻辑错误（交互不符合预期、数据展示有误） | `memory/bugs.md` | `BUG-{序号}` |
| Crash / 应用崩溃 / ANR / OOM | `memory/crashes.md` | `CRASH-{序号}` |
| 难以稳定复现的偶现问题（阶段七进入后） | `memory/hard-problems.md` | `HARD-{序号}` |

文件不存在时先按 [common.md §4.3](common.md) 对应模板初始化，再追加条目。序号在同一文件内单调递增，不跨文件共享。

---

### Step 5：人工升级判定（疑难定位后必执）

完成 Step 1～4 后，按以下标准判断是否需要挂起并人工介入：

> **修复次数计算**：见本阶段顶部规则定义，此处不重复。

| 条件 | 处理方式 |
|------|---------|
| 执行 Step 3 后已有明确根因且有可行方案 | 继续修复，回到阶段六 |
| Step 3 查询无结果，或根因涉及平台 Bug / 接口缺陷 | **挂起任务，通知用户人工介入** |
| 从阶段六起累计修复次数已超过 5 次（含阶段七内修复） | **强制挂起，通知用户人工介入** |

挂起时向用户输出：

```
⚠️ 当前问题已超出自动修复能力范围，建议人工介入。

问题摘要：{BUG-{序号} 标题}
已尝试方案：{列出已执行的修复方向及失败原因}
当前卡点：{根因未明 / 平台限制 / 接口缺陷}
建议下一步：{人工排查方向 / 提交平台 Issue / 联系模块负责人}
```

> 挂起后禁止继续自动修复，等待用户明确指令后再继续。

## 阶段八：功能验收（联调完成后必执）

> 本阶段在阶段六问题修复完成、无新反馈后执行，与"疑难问题定位"无关，任何情况下均不得跳过。

逐条对应需求文档 §4 的交互行为执行验收：

- [ ] 核心主流程（Happy Path）可正常走通
- [ ] 空态 / 加载中 / 错误态 UI 符合设计稿描述
- [ ] 分页加载：触底后正确加载更多，不重复 / 遗漏
- [ ] 多设备断点（sm / md / lg 或对应平台断点）布局无溢出 / 重叠
- [ ] 交互后状态一致性（点击操作后 UI 是否正确刷新）

**验收结果判定**：

- 全部通过 → 进入 `review.md` 进行代码评审
- 存在问题 → 返回阶段六重新修复，修复后重新执行本阶段验收

## 编码中途 PRD 变更处理

> 编码过程中发现 PRD 描述与实际实现存在偏差，或用户提出需求调整时，按本节处理，禁止擅自修改 PRD 或静默忽略差异。

### 触发条件（满足任一即触发）

- 编码时发现 PRD §3/§4/§9 中的描述与平台 API 实际行为不符
- 用户在编码阶段提出与 PRD 不一致的新要求
- 联调时发现设计稿已更新，导致 PRD 描述失效

> **不触发本节的情形**（不视为"偏差"，无需走变更流程）：
> - 代码实现细节（变量命名、函数拆分方式）与 PRD 未提及的实现方式不同
> - 因平台 API 限制导致的等效替代实现（功能一致，仅写法不同）
> - PRD §10 中已标注为待确认且尚未解决的项（应在进入编码前解决，漏掉则按本节逻辑变更处理）

### 处理流程

**Step 1：评估变更影响范围**

| 变更类型 | 判断标准 | 处理方式 |
|---------|---------|---------|
| 局部微调 | 仅影响单个区域样式/文案/边距，不涉及接口和状态结构 | 就地更新 PRD 对应章节，注明变更原因，继续编码 |
| 逻辑变更 | 影响 §4 交互行为、§6 数据模型或 §9 接口 | 必须先暂停编码，更新 PRD 并通知用户确认后再继续 |
| 架构级变更 | 影响 §8 文件结构、新增模块依赖、改变状态管理方式 | 暂停编码，回到 `prd-gen.md` **阶段一-A（方案论证与可行性确认）** 重新论证，论证结论回写调研报告 §8 / PRD §8 等相关章节后再继续编码 |

**Step 2：更新 PRD 文档**

1. 在变更章节顶部追加变更标注：`[v1.x 变更 YYYY-MM-DD]`，说明变更原因
2. 若变更涉及 §10 待确认项，补充新的确认项
3. 更新 `/docs/prd/README.md` 中对应条目的版本号

**Step 3：确认后继续编码**

- 局部微调：更新完成后直接继续，无需用户确认
- 逻辑变更/架构级变更：向用户输出变更摘要并等待确认，确认后继续编码
