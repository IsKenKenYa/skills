# 错误恢复策略

## 恢复机制总览

任何 skill 中断（会话断开、超时、用户中止）后，重新执行同一 skill 即可从断点恢复。

| skill | 恢复机制 | 说明 |
|-------|---------|------|
| `/pipeline import` | 幂等 — 重新执行即可 | 检测已有同名需求，让用户选择覆盖/换名/取消 |
| 项目检测 | 幂等 — 重新执行即可 | 无中间状态，重新分析不丢失信息 |
| 项目初始化 | 幂等 — 已准备好的跳过 | 检测已存在的目录，增量准备 |
| `/native-analyse` | 检测 `tech-spec.md` + `metadata.json` | 发现未确认的 draft 方案，提示继续/废弃/修改 |
| `/native-plan` | 检测 `plan-{platform}.md` + `metadata.json` | 发现未确认的 draft 计划，提示继续/废弃/修改 |
| `/native-coding` | 读取 `coding-progress-{platform}.json` | 从最后完成的子任务断点继续 |
| `/native-build-fix` | 无状态 — 重新执行即可 | 每次独立运行，无需恢复 |

**总原则**：pipeline 的所有状态都持久化在 `.bundle-flow/` 目录中，不依赖会话内存。新会话中执行 `/pipeline` 会自动读取 state.json，展示当前进度并推断下一步。

## 错误分级处理

遇到错误时，按以下分级策略处理，避免一律重试或一律放弃：

| 错误类型 | 信号特征 | 处理策略 |
|---------|---------|---------|
| **瞬时错误** | 网络超时、git clone 抖动、编译服务暂不可用 | 重试 1 次，不改方案 |
| **逻辑错误** | 方案假设不成立、API 行为与预期不符、依赖关系错误 | 回退到上一阶段（如 plan → analyse），修正后重新执行 |
| **认知错误** | 做完发现解决的不是用户的需求、理解偏差 | 回退到 detect/analyse，重新理解需求 |
| **信息不足** | 执行到一半发现缺少关键信息（如缺少接口文档） | 暂停，用工具主动搜索；搜不到再问用户 |

## 反钻牛角尖机制

设置硬性规则防止在死路上越走越深：

| 场景 | 触发条件 | 动作 |
|-----|---------|------|
| brain-storm 轮次过多 | analyse/plan 中 brain-storm 超过 **5 轮**仍未收敛 | 强制总结当前共识，输出 draft 方案供用户确认 |
| 构建修复循环 | `/native-build-fix` 连续 **3 轮**未减少错误数 | 暂停，输出已修复项和剩余错误摘要，请求用户介入 |
| 阶段交互过多 | 单个阶段内用户交互超过 **10 次** | 提示用户「当前阶段交互较多，是否需要调整需求范围或拆分需求？」 |
| 同一方向反复微调 | 连续 2 次修改同一文件的同一区域且未解决问题 | 停下来，切换到本质不同的方案 |

## 上下文蒸馏

每个阶段的产出物顶部包含**决策摘要块**，便于后续阶段快速恢复上下文，无需重新阅读全文：

```markdown
<!-- DECISION_SUMMARY
需求: [一句话描述]
关键决策:
  - [决策1]: [选项] — 理由: [为什么]
  - [决策2]: [选项] — 理由: [为什么]
约束: [关键约束]
上游依赖: [依赖的产出物文件]
-->
```

各阶段的决策摘要内容：

| 阶段 | 摘要包含 |
|------|---------|
| detect | 需求关键词、选中的模块及角色、平台列表 |
| analyse | 核心技术选型、关键设计决策、跨端契约 |
| plan | 子任务依赖关系、关键实现路径、风险点 |
| coding | 已完成的子任务、关键代码变更、遗留问题 |

## 平台隔离线

在多平台 plan/coding 阶段，切换平台时执行上下文隔离：

1. **完成当前平台** → 输出该平台的完整进度摘要
2. **清理平台上下文** → 新平台只加载：tech-spec.md + 新平台的项目信息 + 新平台的 plan（如已有）
3. **不交叉引用** → 平台 A 的实现细节不影响平台 B 的编码决策（除非是共享层）
