# Push Kit Master（推送服务主 Skill）

这是 Push Kit 推送服务的 Master Skill，负责根据开发者需求智能路由到具体的子 Skill 实现。

## Skill 架构

```
push-kit-helper (Master Skill - 大路由)
├── hmos-push-kit-token (获取Push Token - 核心基础能力)
│   ├── 判断项目中是否存在getToken代码
│   ├── 引导开发者开发push token获取功能
│   ├── 向开发者说明token的意义和价值
│   └── 可作为单独接入能力使用
├── hmos-push-kit-notification (发送通知消息)
│   ├── 先检查是否已接入Token（如未接入，引导先接入Token）
│   ├── 发送基础通知消息
│   ├── 配置通知消息样式
│   ├── 配置点击消息动作
│   └── 数据传递与接收
├── hmos-push-kit-voip (推送应用内通话消息/voip)
│   ├── 先检查是否已接入Token（如未接入，引导先接入Token）
│   ├── 语音/视频来电通知
│   ├── 呼叫接听界面配置
│   ├── 应用在前台处理消息
│   └── 申请voip权益
└── hmos-push-kit-background (推送后台消息)
    ├── 先检查是否已接入Token（如未接入，引导先接入Token）
    ├── 创建 PushMessageAbility 接收消息
    ├── 配置 skills 标签
    ├── 应用在前台处理消息
    └── （可选）配置 proxyData 数据库存储
```

## 核心设计理念

### Push Token 的核心地位

**所有推送服务的基础都是获取 Push Token**。无论选择哪种场景化消息（通知、通话、语音），都需要首先完成 Push Token 的获取。

**本 Skill 的设计原则**：
1. 每个场景化消息 Skill 都需要先检查开发者是否已正确获取 Push Token
2. 如果没有 Token 或不确定，需要引导开发者先完成 `hmos-push-kit-token` 场景
3. 这确保了开发者的推送集成工作从一开始就建立在正确的基础上

## 场景化消息权益说明

Push Kit 支持多种场景化消息权益，部分需要申请才能使用：

| 权益名称 | 说明 | 每日推送限制 | 提醒方式 |
|---------|------|------------|---------|
| 即时聊天（IM） | 社交通讯类消息 | 无限制 | 锁屏+铃声+振动 |
| 音视频通话（voip） | 来电通知 | 无限制 | 锁屏+铃声+振动 |
| 出行提醒（TRAVEL） | 出行相关通知 | 无限制 | 锁屏+铃声+振动 |
| 健康数据（HEALTH） | 健康运动通知 | 无限制 | 锁屏+铃声+振动 |
| 工作事项（WORK） | 工作任务提醒 | 无限制 | 锁屏+铃声+振动 |
| 资讯营销（默认） | 活动和内容推荐 | 2-5条/设备/日 | 静默通知 |

> **重要提示**：未申请自分类权益的应用，默认推送的都是资讯营销类消息，会受到"每日每设备 2-5 条"的严格限制！

## 开发流程总览

### Step 1: 开通推送服务（必须）

推送服务需要在 AppGallery Connect 中开通才能使用。这是获取有效 Push Token 的前提。

**操作路径**：AppGallery Connect → 项目 → 应用 → 增长 → 推送服务 → 立即开通

### Step 2: 配置应用签名（必须）

签名配置决定应用能否正常获取 Push Token：
- **调试阶段**：使用调试证书 + 调试 Profile
- **发布阶段**：使用发布证书 + 发布 Profile
- **自动签名**：DevEco Studio 6.0.0+ 支持自动配置

### Step 3: 获取 Push Token（必须）

所有推送的起点，在应用启动时调用 `getToken()`。

### Step 4: 申请场景化消息权益（推荐）

申请对应类型的自分类权益，可以突破每日推送数量限制。

### Step 5: 开发具体功能

根据业务需求，选择对应的场景化消息类型进行开发。

## 常见问题快速排查

| 问题现象 | 可能原因 | 解决方案 |
|---------|---------|---------|
| `getToken()` 返回错误码 1000900010 | 推送服务未开通或签名问题 | 检查推送服务是否开通，重新申请 Profile |
| 收不到推送消息 | Token 过期或签名问题 | 重新获取 Token，检查签名配置 |
| Token 一直获取失败 | 调试设备未注册 | 注册调试设备 |
| 每日推送数量受限 | 未申请自分类权益 | 申请通知消息自分类权益 |
| 推送数量被频控 | 测试阶段未使用 testMessage 参数 | 设置 testMessage 为 true 进行测试 |

## 代码生成规则

生成代码时必须遵循：
1. 使用正确的导入路径：`@kit.PushKit`、`@kit.BasicServicesKit` 等
2. 所有 ArkTS 类型注解必须正确
3. 使用 Promise 处理异步操作
4. 添加适当的错误处理
5. 使用 `hilog` 模块记录日志
6. 生成的代码必须能够直接编译通过
7. 遵守 HarmonyOS 应用开发规范
