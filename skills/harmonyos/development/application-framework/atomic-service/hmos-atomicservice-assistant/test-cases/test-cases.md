# 测试用例 1：新建元服务项目引导

## 场景描述
用户希望从零开始创建一个鸿蒙元服务，不了解元服务与普通应用的区别，需要 AI 代理给出完整的新建流程和关键配置指导。

## 前置条件
- 用户已安装 DevEco Studio 并配置好 HarmonyOS SDK
- 用户没有现成工程，计划新建

## 执行步骤
1. AI 代理识别用户请求关键词"新建元服务"，触发元服务决策树中"新建元服务"分支
2. AI 代理向用户确认业务场景和目标平台（手机/平板/折叠屏）
3. AI 代理给出以下指导：
   - 在 DevEco Studio 新建工程时选择 Atomic Service 模板
   - 在 `app.json5` 中设置 `bundleType: "atomicService"`
   - 说明单分包不超过 2MB、总包不超过 10MB 的限制
   - 提醒只能使用带有 `@atomicservice` 标注的 API
   - 指导配置隐私托管和可信域名

## 预期结果
- AI 代理正确识别"新建元服务"意图，不走"改造现有应用"分支
- 输出中包含 `bundleType: "atomicService"` 的明确说明
- 输出中包含包大小限制（单分包 ≤ 2MB，总包 ≤ 10MB）
- 输出中包含 `@atomicservice` API 限制说明
- 输出中包含隐私托管配置提示
- 提供完整的新建流程步骤，逻辑连贯

## 验证方法
- 检查输出是否包含 `bundleType` 关键配置项
- 检查输出是否提及包大小限制数值（2MB / 10MB）
- 检查输出是否提及 `@atomicservice` 注解限制
- 检查是否有隐私托管或可信域名相关指引
- 检查流程步骤是否完整且符合实际操作顺序

## 备注
新建元服务是最高频的使用场景，AI 代理应能在用户描述模糊时主动询问业务类型，而不是直接给出泛化答案。

# 测试用例 2：将现有应用改造为元服务

## 场景描述
用户已有一个普通 HarmonyOS 应用，希望将其改造为元服务（免安装形态），需要 AI 代理指导改造步骤和注意事项。

## 前置条件
- 用户有一个已存在的 HarmonyOS 应用工程（bundleType 为 app）
- 该应用使用了部分非元服务 API
- 应用存在 ArkWeb 组件用于嵌入 H5 页面

## 执行步骤
1. AI 代理识别用户请求中"改造现有应用"/"迁移到元服务"意图
2. AI 代理触发决策树"改造现有应用"分支
3. AI 代理给出改造检查清单，包含以下 Critical 项：
   - 修改 `app.json5` 的 `bundleType` 为 `"atomicService"`
   - 全量检查并替换非 `@atomicservice` API 调用
   - 将 `ArkWeb` 组件替换为 `AtomicServiceEnhancedWeb`
   - 配置可信域名（网络请求白名单）
   - 配置隐私托管
   - 检查包大小是否满足 ≤ 2MB（单包）/ ≤ 10MB（总包）
   - 静默登录接口替换为 `getQuickLoginAuthCode`
   - 检查跳转规范（不可跳转普通应用）
   - 检查路由是否使用 Navigation + NavPushPathHelper
4. AI 代理同时给出 Warning 级别提醒：睫毛图标规范、Menubar 避让、分享功能、ICP 备案

## 预期结果
- AI 代理正确识别"改造"意图，输出改造专属流程（区别于新建流程）
- Critical 检查项全部出现在输出中
- Warning 提醒至少包含睫毛图标和 Menubar 避让
- `ArkWeb` → `AtomicServiceEnhancedWeb` 替换明确提及
- 静默登录 `getQuickLoginAuthCode` 明确提及

## 验证方法
- 检查输出是否包含 9 个 Critical 合规检查项
- 检查 `AtomicServiceEnhancedWeb` 是否出现在输出中
- 检查 `getQuickLoginAuthCode` 是否出现在输出中
- 检查 Warning 级别提示是否独立列出，与 Critical 有所区分

## 备注
改造场景中最容易遗漏的是 API 兼容性检查和 H5 组件替换，AI 代理应着重强调这两点。

# 测试用例 3：元服务合规检查

## 场景描述
用户已完成元服务开发，即将提交审核，希望 AI 代理对其元服务进行全面的合规检查，确认是否满足上架要求。

## 前置条件
- 用户有一个已开发完成的元服务工程
- 用户提供了部分配置信息（如 `app.json5` 内容片段）
- 用户尚未进行上架提交

## 执行步骤
1. AI 代理识别用户请求中"合规检查"/"上架前检查"/"审核前检查"意图
2. AI 代理输出完整的合规检查清单，分为 Critical（必须修复）和 Warning（建议修复）两级
3. Critical 项包含：
   - `bundleType` 是否为 `"atomicService"`
   - 所有 API 是否带有 `@atomicservice` 标注
   - 可信域名是否配置（网络请求域名白名单）
   - 隐私托管是否配置
   - 静默登录是否使用 `getQuickLoginAuthCode`
   - 包大小是否满足限制（单包 ≤ 2MB，总包 ≤ 10MB）
   - H5 组件是否使用 `AtomicServiceEnhancedWeb`（非 `ArkWeb`）
   - 跳转行为是否符合规范（不跳普通应用）
   - 路由是否使用 Navigation + NavPushPathHelper
4. Warning 项包含：
   - 睫毛图标规范（图标左上角白色弧线标识）
   - Menubar 避让逻辑
   - 分享功能接入
   - ICP 备案（若涉及互联网信息服务）
5. AI 代理对用户提供的配置片段进行针对性分析，指出具体问题

## 预期结果
- AI 代理输出分级合规检查清单（Critical / Warning 分开列出）
- 9 个 Critical 检查项全部覆盖
- 4 个 Warning 检查项全部覆盖
- 若用户提供了配置片段，AI 代理能针对具体内容给出问题定位

## 验证方法
- 统计输出中 Critical 项数量，应 ≥ 9
- 统计输出中 Warning 项数量，应 ≥ 4
- 检查 Critical 和 Warning 是否有明确的视觉区分（标题、分组或图标）
- 验证每个检查项是否有说明和建议操作

## 备注
合规检查是上架前的关键环节，AI 代理必须输出完整清单，不能遗漏 Critical 项，否则可能导致用户审核失败。

# 测试用例 4：高频技术问题速答

## 场景描述
用户在元服务开发过程中遇到多个典型技术问题，希望 AI 代理快速给出解决方案，包括：包大小超限、H5 组件报错、静默登录失败、网络请求被拦截。

## 前置条件
- 用户正在开发元服务，已有工程
- 用户遇到具体的运行时或编译时错误

## 执行步骤

### 子场景 4-1：包大小超限
1. 用户描述："我的元服务编译后单个分包超过 3MB，怎么处理？"
2. AI 代理识别"包大小超限"问题，触发高频问题速答
3. AI 代理给出：单分包 ≤ 2MB 限制说明、分包拆分建议、资源压缩建议、使用网络资源替代本地资源建议

### 子场景 4-2：H5 组件报错
1. 用户描述："我用 ArkWeb 加载 H5 页面，在元服务里报错"
2. AI 代理识别"H5 组件"问题
3. AI 代理明确指出：元服务不支持 `ArkWeb`，必须使用 `AtomicServiceEnhancedWeb` 组件

### 子场景 4-3：静默登录失败
1. 用户描述："元服务里调用华为账号登录接口，提示接口不支持"
2. AI 代理识别"静默登录"问题
3. AI 代理给出：使用 `getQuickLoginAuthCode` 接口说明，以及典型代码调用示例

### 子场景 4-4：网络请求被拦截
1. 用户描述："元服务访问我的后端 API 域名被拦截了"
2. AI 代理识别"可信域名"问题
3. AI 代理给出：在华为开发者联盟配置可信域名的操作步骤

## 预期结果
- 子场景 4-1：输出包含"2MB"限制和至少一种减包方案
- 子场景 4-2：输出明确包含 `AtomicServiceEnhancedWeb`，并指出 `ArkWeb` 不可用
- 子场景 4-3：输出包含 `getQuickLoginAuthCode` 接口名称
- 子场景 4-4：输出包含"可信域名"配置操作路径

## 验证方法
- 每个子场景的关键词（接口名、组件名、数值）必须出现在对应输出中
- AI 代理不应将不同子场景的解决方案混淆
- 回答应简洁直接，高频问题不需要冗长铺垫

## 备注
高频问题速答要求 AI 代理能精准识别问题类型并快速给出答案，不应引导用户查看文档而不给出实质性解答。

# 测试用例 5：元服务上架审核失败处理

## 场景描述
用户提交元服务到华为应用市场后，收到审核驳回通知，驳回原因涉及多个问题，需要 AI 代理帮助分析原因并给出修复方案。

## 前置条件
- 用户已提交元服务审核
- 收到审核驳回通知，包含以下驳回原因（模拟）：
  1. "元服务图标不符合规范"
  2. "应用中存在不支持的 Web 组件"
  3. "隐私政策未托管至华为平台"

## 执行步骤
1. 用户将驳回原因告知 AI 代理，请求帮助分析和修复
2. AI 代理识别"上架审核失败"意图，触发对应决策树分支
3. AI 代理逐条分析驳回原因：
   - 图标不符合规范：说明睫毛图标规范（左上角白色弧线标识），给出设计要求
   - Web 组件不支持：指出需将 `ArkWeb` 替换为 `AtomicServiceEnhancedWeb`
   - 隐私政策未托管：指导在华为开发者联盟完成隐私托管配置步骤
4. AI 代理给出修复优先级建议和重新提交前的完整检查建议

## 预期结果
- AI 代理逐条响应三个驳回原因，不遗漏任何一条
- 图标问题：输出包含睫毛图标规范说明
- Web 组件问题：输出包含 `AtomicServiceEnhancedWeb` 替换方案
- 隐私托管问题：输出包含华为开发者联盟隐私托管操作路径
- AI 代理建议用户修复后重新执行合规检查清单再提交

## 验证方法
- 检查三个驳回原因是否均有针对性回应
- 检查 `AtomicServiceEnhancedWeb` 是否出现在输出中
- 检查隐私托管操作路径是否具体可操作
- 检查是否有"重新提交前检查"的建议

## 备注
审核失败场景中，AI 代理应展现出"逐条分析 + 给出修复 + 预防再次失败"的完整处理思路，而不仅仅是描述问题。

# 测试用例 6：元服务路由与跳转规范指导

## 场景描述
用户在元服务中实现页面跳转功能时，不确定应使用哪种路由方式，以及跳转到其他应用时的合规规范。

## 前置条件
- 用户已有元服务工程，正在实现多页面导航
- 用户曾在普通 HarmonyOS 应用中使用 `router` 模块进行页面跳转

## 执行步骤
1. 用户询问："元服务页面跳转应该用什么方式？能不能跳到其他 App？"
2. AI 代理识别路由和跳转规范相关问题
3. AI 代理给出路由方案说明：
   - 推荐使用 Navigation 组件 + NavPushPathHelper 实现页面导航
   - 说明不推荐使用旧版 `router` 模块的原因
4. AI 代理给出跳转规范说明：
   - 元服务不可跳转至普通应用（非元服务）
   - 元服务间跳转需遵循相关规范
   - 跳转行为违规是审核被拒的常见原因之一

## 预期结果
- 输出中明确推荐 Navigation + NavPushPathHelper 路由方案
- 输出中明确说明不可跳转至普通应用
- 输出中提及跳转违规会导致审核失败
- 路由代码示例或配置说明清晰

## 验证方法
- 检查 `Navigation` 和 `NavPushPathHelper` 是否出现在输出中
- 检查是否有"不可跳转普通应用"的明确说明
- 检查输出是否与合规检查清单中的跳转规范项一致

## 备注
路由规范和跳转限制是元服务区别于普通应用的重要特性，AI 代理需清晰区分两者，避免给出错误的跳转方案。
